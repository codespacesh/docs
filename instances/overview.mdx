An **instance** is a fully isolated Coder deployment managed by Codespace. Each instance has its own infrastructure, domain, and team.

## What's inside an instance

When you create an instance, Codespace provisions:

| Resource | Description |
|----------|-------------|
| **Kubernetes namespace** | Isolated container environment |
| **PostgreSQL database** | Dedicated database for Coder state |
| **Coder Helm release** | The Coder application itself |
| **TLS certificate** | Automatic HTTPS via Let's Encrypt |
| **DNS record** | `your-subdomain.codespace.sh` |

## Instance lifecycle

Instances progress through these states:

```
Pending → Provisioning → Running
                       ↘ Failed → (Retry) → Provisioning → Running
Running → Deleting → Deleted
```

See [Instance statuses](/reference/instance-statuses) for details on each state.

## Owner vs member access

| Action | Owner | Member |
|--------|-------|--------|
| Create workspaces | Yes | Yes |
| View instance details | Yes | Yes |
| Manage team members | Yes | No |
| Configure settings | Yes | No |
| Delete instance | Yes | No |
| Upgrade Coder version | Yes | No |

## Limits

Each instance has resource limits based on your [subscription plan](/reference/subscription-plans):

- **Workspaces per user** — how many workspaces each team member can create
- **Total workspaces** — maximum across all members
- **CPU hours** — compute time per billing period
- **Memory** — RAM allocation per billing period
- **Storage** — disk space allocation
